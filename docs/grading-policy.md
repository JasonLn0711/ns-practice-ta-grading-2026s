# Grading Policy

## Core Rules

- Fairness over cleverness.
- Consistency over improvisation.
- Fast repeated execution over one-time perfection.
- Clear logs over vague memory.
- Separate objective checks from subjective grading.
- Surface policy ambiguity instead of silently inventing rules.

## TA Operating Policy

- Use the same rubric for every student.
- Do not invent per-student standards.
- Log exceptions once in `grading/<hw>/review_notes/`.
- Escalate suspicious plagiarism or unusual similarity to the instructor.
- Separate attendance notes from grading notes.
- Do not keep sensitive student data in random notes.
- Use feedback tags to speed up repeated feedback.
- When unsure, mark `manual_review_needed=yes` and write the smallest clear reason.

## Objective Checks vs Manual Review

Objective checks can include:

- required file exists
- archive can be opened
- notebook or script is present
- output plots or screenshots are present
- score row has numeric totals

Manual review must decide:

- code correctness
- whether outputs are reproducible
- whether the method matches the assignment
- whether explanation quality deserves partial credit
- whether a late or corrupted submission should receive an exception

## Exception Log Rule

When an exception appears, log:

- date
- student identifier
- issue
- action taken
- whether instructor confirmation is needed

Do this once. Do not scatter the same exception across daily notes, chats, and score comments.
