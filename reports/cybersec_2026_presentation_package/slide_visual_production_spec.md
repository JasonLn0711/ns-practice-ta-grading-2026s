# CYBERSEC 2026 Slide Visual Production Spec

Purpose: give the slide editor a concrete production brief for each slide: exact screen copy, composition, visual treatment, reveal behavior, and scoring reason.

This file is intentionally visual and operational. It should be used when editing the `.pptx`, checking exported PDFs, or reviewing slides on a projector.

## 1. Global Visual System

### Style Direction

Premium healthcare-security briefing: restrained, technical, diagram-led, high-contrast, and evidence-oriented. The deck should look like a serious product-risk and regulatory-evidence talk, not a class report, startup pitch, or decorative AI theme.

### Composition Rules

| Rule | Production Decision | Scoring Protected |
| --- | --- | --- |
| One dominant object per slide | Each slide should have one phrase, diagram, process, contrast, or roadmap as the visual center | Visual Design of Slides, Structure and Narrative Design |
| Text must be skimmable in three seconds | Use one large title, one short message, and no paragraph blocks | Visual Design of Slides, Stage Rhythm and Time Control |
| Diagrams carry logic | Use ladders, stacks, traces, and flows for architecture/evidence/process | Content Depth and Value |
| Color has meaning | Accent only risk, decision, active layer, or action path | Visual Design of Slides |
| No decorative AI wallpaper | Avoid random circuits, glowing brains, stock hacker imagery, and abstract gradients without semantic function | Stage Presence, Audience Impact |

### Typography And Density

- Title: one clear claim, not a label when possible.
- Supporting text: maximum one sentence.
- Labels: short noun phrases; avoid full sentences inside diagrams.
- Minimum practical font size for projected labels: keep labels readable from the back of a conference room.
- If a diagram needs more than six labels, group labels into layers or reveal progressively.

### Palette Logic

- Base: near-black or clean white, depending on official template constraints.
- Primary accent: medical/security cyan or blue-green for evidence/action.
- Risk accent: red only for disruption, exposure, or decision pressure.
- Neutral: gray for background layers and inactive architecture.
- Do not let the deck become a single-color monotone; use accent sparingly and semantically.

## 3. Visual QA Checklist

Run this check on the exported PDF before rehearsal:

- [ ] Slides 3, 6, 17, 20, and 23 are visually distinctive from the rest.
- [ ] Slides 7-12 do not look like regulatory text dumps.
- [ ] Slides 13-16 share a coherent scale pattern but do not look identical.
- [ ] Slide 20's `Decision` node is visually dominant.
- [ ] No slide uses decorative AI imagery that does not explain the concept.
- [ ] No slide contains private architecture, proprietary code, exploit detail, private hospital/client detail, or student data.
- [ ] Every slide can be understood at headline level within three seconds.
- [ ] The final slide contains no new content after the closing principle.

## 4. If The Deck Feels Too Dense

Apply this cut order:

1. Remove secondary labels before removing diagrams.
2. Remove explanatory footers before removing core screen copy.
3. Collapse repeated labels in slides 13-16 into a shared pattern.
4. Move advanced details to Q&A, especially confidential computing, attestation, specific standards, and tool choices.
5. Preserve memory slides 3, 6, 17, 20, and 23.
